## 1. Product Overview

LennyHub RAG is a production-ready retrieval-augmented generation system built on top of 297 transcripts from Lenny's Podcast — a show featuring prominent product managers and growth practitioners. It lets you ask natural-language questions across the entire podcast catalog and get AI-synthesized answers grounded in actual episode content. The system ships with a one-command setup script, a Streamlit web UI, a CLI query tool, and an interactive network graph showing connections between 544 podcast guests and their topics.

---

## 2. Problem It Solves

Lenny's Podcast has nearly 300 episodes packed with product, growth, and strategy insight — but that knowledge is locked inside individual audio/transcript files with no cross-episode search. Without this system, a PM trying to find "what do guests say about activation metrics?" would have to manually scan transcripts, rely on fuzzy episode memory, or use a generic web search that misses nuance. LennyHub RAG ingests all transcripts into a local vector + knowledge-graph database so any insight can be surfaced in seconds with a single question.

---

## 3. Who It's For

Product managers, growth practitioners, and startup founders who rely on Lenny's Podcast as a reference resource and want to query across all 297 episodes without manually reading transcripts.

## 4. How It Works (Non-Technical)

- You run a one-command setup that reads all podcast transcripts and builds a searchable knowledge base locally on your machine.
- You ask questions in plain English — either through a web app or the command line.
- The system finds the most relevant transcript passages, extracts relationships between guests and topics, and generates a synthesized answer with supporting context.
- A separate graph viewer lets you explore how 544 podcast guests are connected to each other and to recurring themes.

---

## 5. Architecture Overview

- **Ingestion pipeline** (`setup_rag.py`): reads `.txt` transcript files, sends them through RAGAnything → OpenAI embeddings → Qdrant (vector store) and LightRAG (entity/relationship graph). Supports sequential or parallel (up to 10 workers) processing.
- **Storage layer**: Qdrant runs as a local server on port 6333 — no Docker required. LightRAG writes a knowledge graph alongside it capturing people, companies, and concepts.
- **Query layer** (`query_rag.py`): exposes four search modes — `hybrid` (default), `local` (entity-focused), `global` (relationship-focused), and `naive` (pure vector). Queries are async via `RAGAnything.aquery()`.
- **UI layer** (`streamlit_app.py`): three-tab Streamlit dashboard (Query / Statistics / Transcripts). Spawns queries in a subprocess to avoid Python async event-loop conflicts with Streamlit's own runtime. Auto-starts Qdrant if it isn't running.

## 6. Key Technologies

| Technology | Role |
|---|---|
| RAGAnything ≥1.2.9 | Core orchestration framework that ties ingestion, retrieval, and LLM generation together |
| LightRAG ≥1.4.9 | Extracts entities and relationships from transcripts into a queryable knowledge graph |
| Qdrant ≥1.16 | Local vector database storing transcript embeddings; chosen specifically to avoid Docker |
| OpenAI GPT-4o-mini | LLM that generates final answers from retrieved context |
| text-embedding-3-small | Embedding model that converts text chunks into vectors for semantic search |
| Streamlit ≥1.28 | Web UI for querying, browsing transcripts, and viewing system stats |
| nest-asyncio | Patches Python's event loop so async RAG calls work inside Streamlit's synchronous runtime |

## 7. Notable Design Decisions

- **No Docker for the vector DB**: Qdrant is run as a local binary rather than a container, lowering the setup barrier significantly — but it also means the server management (start/stop, port conflicts) is handled ad-hoc inside the Python scripts rather than by a container orchestrator.
- **Subprocess isolation for Streamlit queries**: Rather than resolving the async/sync conflict architecturally, the app spawns `query_rag.py` as a child process. This works but means query state isn't shared in-process and adds subprocess startup overhead per query.
- **Cost-first defaults**: The defaults (GPT-4o-mini, text-embedding-3-small, response caching) are optimized for low cost (~$7.20 total for all 297 transcripts; $0.001–0.01 per query) rather than highest quality — a deliberate trade-off for a research/exploration tool.

## 8. How to Run It

1. Clone the repo: `git clone https://github.com/traversaal-ai/lennyhub-rag`
2. Install dependencies: `pip install -r requirements.txt`
3. Create `.env` and add `OPENAI_API_KEY=sk-...`
4. Run setup: `python setup_rag.py` (choose quick / partial / full mode; use `--parallel` for 5–10x faster indexing)
5. Launch the app: `streamlit run streamlit_app.py`

---

## 9. What's Missing or Incomplete

- **No authentication**: The Streamlit app has no login or access control — anyone who can reach the port can query the system.
- **OpenAI-only**: No support for Anthropic, local (Ollama), or other LLM/embedding providers; switching would require code changes throughout `setup_rag.py` and `query_rag.py`.
- **Graph viewer is disconnected**: The knowledge graph explorer (`serve_graph.py`) is a separate process, not integrated into the main Streamlit dashboard — users have to launch and find it independently.
